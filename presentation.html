نظام مراقبة الخوادم 🚀
Server Alert Manager

Built with Bash & Python

"نظام ذكي، خفيف، وآمن لمراقبة موارد السيرفر وإرسال التنبيهات"

📌 إيه هو المشروع ده؟

الهدف الأساسي هو إننا نعمل Monitoring لموارد السيرفر الحساسة زي مساحة الديسك Disk Space والرامات RAM.

ولما الاستهلاك يعلى عن الحد المسموح، السيستم بيبعت Automated Alert (تنبيه أوتوماتيكي) على الإيميل عشان الـ SysAdmin يتدخل قبل ما السيرفر يقع.

🛠️ التقسيم المعماري (Architecture)

عشان نكتب كود نظيف، طبقنا مبدأ Separation of Concerns (فصل المهام).

  • الجزء الأول (المراقب): سكريبت Bash خفيف جداً بيتكلم مع نظام التشغيل عشان يقرأ الموارد من غير ما يستهلك الـ CPU.
  • الجزء التاني (المرسل): سكريبت Python قوي في الشبكات، بياخد الرسالة من الـ Bash ويبعتها لـ Gmail عن طريق اتصال مشفر.

❌ ليه معملناش MTA على السيرفر؟

طريقة Postfix القديمة

كان ممكن نستخدم أمر mail في لينكس، بس ده بيحتاج نسطب Postfix MTA.

  • بيستهلك موارد السيرفر (RAM).
  • ممكن الهاكرز يستغلوه كـ Open Relay ويبعتوا منه Spam.
  • محتاج إعدادات DNS معقدة (SPF, DKIM).

طريقة Python الذكية

استخدمنا مكتبة smtplib في بايثون.

  • السيرفر بيشتغل كـ (Client) بيكلم جوجل مباشرة.
  • الإيميل بيوصل 100% للـ Inbox.
  • مفيش بورتات غريبة بتتفتح في الـ (Firewall).

🔍 تفصيل الـ Bash Script

استخدمنا أدوات اللينكس الأساسية Coreutils عشان سرعتها الخرافية.


# جلب مساحة الديسك كرقم صافي
CURRENT_DISK=$(df / | grep / | awk '{ print $5}' | sed 's/%//g')

# جلب الرامات الفاضية بالميجا بايت
FREE_RAM_MB=$(free -m | awk 'NR==2{print $4}')

شرح الكود: أمر df / بيجيب جدول المساحة، awk بيقص العمود رقم 5، و sed 's/%//g' بيمسح علامة % عشان نتحصل على رقم نقارنه في الـ If Condition.

📨 تفصيل الـ Python Script

بمجرد ما الـ Bash يكتشف خطر، بينادي البايثون ويديله الرسالة.


from email.message import EmailMessage
import smtplib, os

msg = EmailMessage()
msg.set_content(alert_payload)
msg['Subject'] = 'CRITICAL: Server Alert'

# Secure SSL connection on Port 465
with smtplib.SMTP_SSL('smtp.gmail.com', 465) as smtp:
    smtp.login(os.environ.get('GMAIL'), os.environ.get('APP_PASS'))
    smtp.send_message(msg)

استخدمنا كلاس EmailMessage عشان نعمل Formatting صح للهيدرز (To, From, Subject)، ده بيمنع جوجل إنها تصنفنا كـ Spam.

🛡️ تأمين النظام (Security Practices)

  • منع الـ Hardcoding: ممنوع نكتب الباسورد جوه الكود! حطيناه في ملف مخفي اسمه .env عشان الكود يقرأه وقت التشغيل بس.
  • كلمات مرور التطبيقات (App Passwords): جوجل منعت الـ Basic Auth، فعملنا App Password مخصص للسكريبت.. لو اتسرق مش هيقدر يفتح جوجل درايف بتاعك، آخره يبعت إيميل وبس.

⚙️ ملف الإعدادات (Configuration)

لمبدأ الـ Maintainability، عملنا ملف monitor.conf:

  • MAX_DISK_USAGE=80
  • MIN_FREE_RAM=500

أي SysAdmin يقدر يغير التنبيهات من الإعدادات من غير ما يلعب في الـ Code الأساسي ويبوظه.

🔎 كيف بنيت فهمي؟ (Proof of Research)

1. قرار استخدام بورت 465 (SSL)

"class smtplib.SMTP_SSL(host='', port=0...)
An SMTP_SSL instance behaves exactly the same as instances of SMTP. SMTP_SSL should be used for situations where SSL is required from the beginning of the connection... it creates a connection which is secured from the start."

كيف فهمتها وبنيت عليها الكود؟

قريت السطر ده في توثيق بايثون الرسمي، وفهمت منه إن بورت 465 (Implicit SSL) أأمن بكتير من بورت 587 (STARTTLS) لأنه بيشفر الاتصال فوراً من قبل حتى ما نبعت كلمة السر. وبناءً عليه صممت الكود بتاعي باستخدام دالة SMTP_SSL() لضمان أقصى حماية لبيانات السيرفر.

🔥 مستوى متقدم (Advanced SysAdmin)

1. الأتمتة (Automation - Crontab)

عشان النظام يشتغل لوحده، السكريبت متصمم إنه يشتغل من خلال الـ Crontab.


*/5 * * * * /home/user/gmail_alert_manager/server_monitor.sh

السطر ده بيخلي اللينكس يشغل السكريبت أوتوماتيك كل 5 دقايق في الخلفية، بحيث يراقب السيرفر 24 ساعة بدون أي تدخل بشري.

🔥 مستوى متقدم (Advanced SysAdmin)

2. حماية الـ Secrets (File Permissions)

إزاي نضمن إن مفيش مستخدم تاني على السيرفر يقرأ الباسورد من ملف .env؟


chmod 600 .env

أمر chmod 600 بيخلي صاحب الملف فقط (Owner) هو اللي يقدر يقرأه. أي مستخدم آخر لو حاول يعمله cat هيطلعله Permission Denied.

🔥 مستوى متقدم (Advanced SysAdmin)

3. التوسع (Scalability & Big Picture)

هل لو عندنا 100 سيرفر، هنحط السكريبت ده عليهم كلهم؟

الإجابة: لأ طبعاً! السكريبت ده حل عبقري لـ Stand-alone servers لأنه Agentless وخفيف جداً.

لكن لو عندنا 100 سيرفر، هنبني Monitoring Infrastructure ضخمة زي Prometheus يسحب الداتا من السيرفرات، و Grafana تعرضها وتبعت التنبيهات.

🚀 خريطة التطوير (Future Enhancements)

1. مراقبة حالة الخدمات (Service Uptime)

النظام الحالي بيراقب الـ Hardware، لكن يمكن تطويره لمراقبة الخدمات الحساسة.

التنفيذ المقترح: إضافة أمر systemctl is-active nginx في الباش سكريبت. في حالة سقوط خادم الويب، يتم إرسال تنبيه فوري قبل أن يتأثر المستخدمون.


2. مراقبة إجهاد المعالج (CPU Load Average)

التنفيذ المقترح: استخدام أمر uptime لقراءة الحمل (Load Average) لآخر 5 دقائق لاكتشاف هجمات الـ DDoS أو العمليات المعلقة.

🚀 خريطة التطوير (Future Enhancements)

3. إيميلات احترافية (HTML Email Templates)

لتوفير واجهة مستخدم أفضل (Enterprise-level) داخل الإيميل.

التنفيذ المقترح: تحديث كود البايثون لدعم إرسال MIMEText('html')، بحيث يصل التنبيه في شكل جداول منسقة وألوان (أحمر للخطر، أخضر للعودة للعمل بشكل طبيعي).


4. الربط بأنظمة التذاكر (Auto-Ticketing API)

التنفيذ المقترح: استخدام مكتبة requests في بايثون لضرب Webhook خاص بنظام الـ Jira أو Trello لفتح تذكرة صيانة (Maintenance Ticket) أوتوماتيكياً بمجرد اكتشاف عطل.

🚀 خريطة التطوير (Future Enhancements)

5. حل إرهاق التنبيهات (Alert Fatigue)

المشكلة: لو الديسك وصل 90%، والـ Crontab بيشتغل كل 5 دقائق، السكريبت هيبعت 288 إيميل في اليوم! ده بيعمل إزعاج شديد للـ SysAdmin وممكن يُصنف كـ Spam.

الحل المقترح (State Management): إضافة ميزة الـ Lock Files.

بمجرد إرسال أول إيميل، نقوم بإنشاء ملف وهمي مثل /tmp/disk_alert.lock. في المرة القادمة، يتأكد السكريبت من وجود الملف؛ إذا كان موجوداً يصمت السكريبت ولا يرسل إيميلات أخرى، حتى تنخفض المساحة فيقوم بحذف الـ Lock File ليعود للعمل الطبيعي.
